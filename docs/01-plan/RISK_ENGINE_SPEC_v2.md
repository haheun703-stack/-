# 퀀트봇 리스크 엔진 설계 지시서 v2 — "미니 알라딘"

> 이 문서는 Claude Code에게 그대로 전달하는 구현 스펙이다.
> 기존 퀀트봇 레포에 `risk/` 모듈을 추가하고, execution 경로에 사전 게이트를 강제 삽입한다.
>
> **★트랙 구분 (6/11 명시)**: 이 문서는 fx-liquidity 지시서와 **별개의 신규 트랙**이다.
> 6/11 확립한 쓰기 3원칙(안전한 기본값 + 레코드 자기상태 선언 + 게이트 분리)의 확장판 —
> "쓰기는 명시적 통과 없이 일어나지 않는다"를 주문 경로로 끌어올린 것.
> ★freeze 주의: Phase 1의 "execution 게이트 토큰 강제 삽입"은 **실주문 경로 수정**이므로,
> 적대검증·테스트로 "게이트 우회 주문이 코드상 불가능"을 증명한 뒤에만 머지한다.
> 게이트는 주문을 *더 막는* 방향이라 freeze 정신과 양립하나, 경로 변경 자체는 신중히.
>
> **★freeze 해제 룰 (6/11 사장님 확정)**: 이 엔진(Phase 1)이 깔리기 전엔 unfreeze 금지.
> "급하지 않다. 대신 freeze 해제의 전제조건이다." Phase 1a(킬스위치+sizing+게이트 로직)는 freeze
> 완전 양립 구간, Phase 1b(execution 배선)는 unfreeze 직전. **체크리스트 = [unfreeze-checklist.md](unfreeze-checklist.md)**.
> ★킬스위치 모의발동은 freeze 중이 가장 안전(실주문0=부작용 없는 유일 창) → 6/12 직후로 너무 미루지 말 것.

## 0. 설계 철학 (코드 주석 최상단에 박을 것)

1. **수익은 결과, 생존이 목표.** 이 엔진의 목적은 수익 극대화가 아니라 "어떤 상황에서도 계좌가 죽지 않는 구조"다.
2. **리스크 관리는 사후 보고서가 아니라 사전 게이트.** 주문은 게이트를 통과해야만 나간다. 게이트를 우회하는 코드 경로는 존재해서는 안 된다.
3. **모든 모델은 틀린다.** VaR, 팩터, 시뮬레이션이 못 잡는 충격은 반드시 온다. 그래서 마지막 층은 모델이 아니라 하드 룰(킬스위치)이다.
4. **노출(exposure)이 아니라 리스크(risk)를 본다.** 1천만원 포지션이 둘 있어도 손절폭이 다르면 리스크는 다르다.

## 1. 아키텍처 — 4겹 방어선

```
L1  트레이드 레벨   리스크 예산 사이징, 손절, 갭 보정, 유동성 한도
L2  포트폴리오 레벨  팩터 노출, VaR/ES, Component VaR, 사전 게이트
L3  레짐 레벨      스트레스 테스트, 드로다운 사다리, 변동성 타겟팅, 크라우딩 모니터
L4  킬스위치       모델 무관 하드 룰. 별도 프로세스. 수동 해제만 가능.
```

디렉토리 구조:

```
risk/
├── __init__.py
├── config.py            # 모든 한도/파라미터 단일 정의 (9장 테이블)

├── factor_exposure.py   # L2: EWMA 가중 팩터 회귀
├── var_engine.py        # L2: FHS VaR/ES + t-분포 몬테카를로
├── component_var.py     # L2: 포지션별 리스크 기여도
├── correlation.py       # L2: 평시/스트레스 이중 상관행렬
├── pre_trade_gate.py    # L2: 사전 게이트 (PASS/RESIZE/REJECT)
├── stress_test.py       # L3: 역사 재생 + 가상 시나리오
├── drawdown_ladder.py   # L3: 드로다운 기반 노출 사다리
├── vol_targeting.py     # L3: 포트폴리오 변동성 타겟팅
├── crowding.py          # L3: 동질화/크라우딩 모니터
├── var_backtest.py      # 모델 검증: VaR 초과 기록 + Kupiec 검정
└── report.py            # 텔레그램 리스크 브리핑 포맷

├── monitor.py
└── state.json           # 발동 상태 영속화 (재부팅에도 유지)
```

## 2. L1 — 트레이드 레벨

### 2.1 리스크 예산 사이징 (`sizing.py`)

원칙: **포지션 크기는 신호 강도가 아니라 손절폭이 결정한다.**

```
R           = 계좌자산 × risk_per_trade        # 초기 0.5%, 최대 1.0%
stop_price  = max(진입가 - 2.0 × ATR(14), 수급 이탈가)   # 둘 중 진입가에 가까운 쪽
stop_width  = (진입가 - stop_price) / 진입가
size_krw    = R / stop_width
size_krw    = min(size_krw, 계좌자산 × max_single_weight)   # 종목 캡 우선
```

- 신호 강도(점수)는 사이즈가 아니라 **진입 여부와 우선순위**에만 사용한다.
- `risk_per_trade`는 백테스트로 "최적화"하지 않는다. 생존 제약이지 수익 파라미터가 아니다.

### 2.2 한국 시장 갭 리스크 보정

한국 장은 미국 장이 끌고 간다. 손절의 상당수는 장중이 아니라 **시초 갭**에서 발생하므로, 장중 손절 주문은 보장 수단이 아니다.

```
gap_worst    = 해당 종목 최근 252일 중 최대 하방 오버나이트 갭 (종가→시가)
effective_stop_width = max(stop_width, gap_worst)
size_krw     = R / effective_stop_width        # 갭 기준으로 사이즈 재계산
```

추가 하드 제약 — **하한가 생존 조건**:

```
max_single_weight ≤ daily_kill_limit / 0.30
# 예: 일일 킬스위치 -3.5%라면, 단일 종목이 하한가(-30%) 가도
#     계좌 손실이 킬스위치 한도를 넘지 않는 비중 = 11.6% → 12%로 캡
```

### 2.3 유동성 한도

```
size_krw ≤ ADV20(20일 평균 거래대금) × 0.05
# 시장 충격 없이 하루 안에 전량 청산 가능해야 한다
```

## 3. L2 — 포트폴리오 레벨

### 3.1 팩터 노출 (`factor_exposure.py`)

팩터 세트 (한국 개인 퀀트 스케일에 맞게 압축):

| 팩터 | 프록시 데이터 |
|---|---|
| 시장 | KOSPI200 일별 수익률 |
| 중소형 | KOSDAQ 일별 수익률 (KOSPI 잔차) |
| 환율 | USD/KRW 일별 변화율 |
| 반도체 | KRX 반도체 지수 (또는 SOX, 시장 잔차) |
| 금리 | 국고채 3년 금리 일별 변화 |

방법론 — **단순 60일 회귀의 함정을 보정**:

1. EWMA 가중 회귀 (halflife = 60일)로 베타 추정 — 최근성 반영
2. 252일 단순 베타도 병행 계산. 두 베타의 괴리가 50% 이상이면 "베타 불안정" 경고
3. 종목별 베타 × 비중 합산 → 포트폴리오 팩터 노출 벡터

출력 예: `KOSPI +1.3 | KOSDAQ +0.2 | FX -0.4 | 반도체 +0.9 | 금리 -0.1`
→ 리포트에 한 줄 해석 자동 생성: *"사실상 반도체 1배 레버리지 베팅 상태"*

### 3.2 VaR/ES — Filtered Historical Simulation (`var_engine.py`)

**순수 히스토리컬 VaR 금지.** 최근 구간이 저변동이면 위험을 체계적으로 과소평가한다.

FHS 절차:

```
1. 종목별 EWMA 변동성 σ_t 계산 (λ = 0.94, RiskMetrics 표준)
2. 과거 수익률 표준화:  z_i = r_i / σ_i          (각 시점의 당시 변동성으로 나눔)
3. 현재 변동성으로 리스케일:  r̃_i = z_i × σ_now
4. 현재 비중 벡터 w를 곱해 가상 포트폴리오 손익 분포 생성 (lookback 750일)
5. 분위수 추출: VaR95, VaR99, ES95 (= 하위 5% 평균)
```

보조 엔진 — 멀티변량 t-분포 몬테카를로:

```
- 상관행렬 + EWMA 변동성으로 다변량 t-분포 (df=6) 시뮬레이션 10,000회
- FHS VaR와 MC VaR의 괴리가 30% 이상이면 경고 출력 (모델 불일치 신호)
```

**오버나이트 VaR 분리 측정** (한국 특수성):

```
- 일별 수익률을 (종가→시가) 갭 수익률과 (시가→종가) 장중 수익률로 분해
- 각각 별도 VaR 산출. "오버나이트 VaR95 = -2.1%" 형태로 리포트
- 오버나이트 VaR가 1일 VaR의 60%를 넘으면 → 장 마감 전 노출 축소 권고 플래그
```

### 3.3 Component VaR (`component_var.py`)

총 VaR을 포지션별 기여도로 분해한다 (한계 VaR × 비중).

```
출력: "종목 A: 비중 8% / VaR 기여 31%" — 비중과 리스크 기여의 불일치를 드러내는 것이 목적
```

게이트(3.5)와 일일 리포트에 사용. 기여도 1위 포지션이 40%를 넘으면 경고.

### 3.4 이중 상관행렬 (`correlation.py`)

**상관관계는 위기 때 1로 수렴한다.** 평시 상관 하나로 분산을 평가하면 안 된다.

```
ρ_normal  = 최근 252일 상관행렬 (EWMA 가중)
ρ_stress  = ρ_normal + 0.5 × (1 - ρ_normal)      # 1 방향 슈링크
# 또는: 과거 위기 구간(VKOSPI 상위 10% 일자)만 모아서 별도 추정 — 데이터 충분하면 이쪽 우선
```

- 평시 VaR: ρ_normal 사용
- **스트레스 VaR**: ρ_stress + 변동성 1.5배 적용 버전
- 게이트는 **둘 중 나쁜 값** 기준으로 판정한다

### 3.5 사전 게이트 (`pre_trade_gate.py`) — 이 모듈이 엔진의 심장이다

**execution 모듈은 이 게이트의 PASS 토큰 없이는 주문을 낼 수 없도록 구조적으로 강제한다.**
(주문 함수의 필수 인자로 `gate_result` 객체를 요구하고, 서명/타임스탬프 검증)

신규 주문 요청 시, **신규 포지션을 포함한 가상 포트폴리오 전체를 재계산**한 뒤 순서대로 체크:

```
G1  포트폴리오 1일 VaR95      ≤ 2.5%        (config: var_limit)
G2  스트레스 VaR95            ≤ 4.0%        (config: stress_var_limit)
G3  단일 종목 비중            ≤ 12%
G4  단일 섹터 비중            ≤ 30%
G5  상관 클러스터: 기존 보유와 ρ ≥ 0.8인 종목들은 합산하여
    "유효 단일 포지션"으로 간주 → G3 한도를 합산 기준으로 적용
G6  유동성: ADV20의 5% 이내
G7  신규 포지션의 Component VaR 기여 ≤ 25%
G8  드로다운 사다리 상태 확인 (4.2) — 현재 단계의 노출 한도 내인가
```

판정 로직:

```
PASS    : 전 항목 통과 → 주문 진행
RESIZE  : G1/G2/G3/G7 위반 시 사이즈를 비례 축소하며 재검사 (최대 5회 반복)
          → 한도 내로 들어오면 축소된 사이즈로 PASS
          → 최소 거래 단위 이하로 떨어지면 REJECT
REJECT  : G4/G5/G6/G8 위반 또는 RESIZE 실패 → 주문 거부 + 사유 로그
```

**모든 게이트 호출은 입력/판정/사유를 JSON 로그로 영구 기록한다** (감사 추적). 게이트 로그가 없는 체결이 발견되면 그것 자체가 Critical 버그다.

## 4. L3 — 레짐 & 스트레스 레벨

### 4.1 스트레스 시나리오 라이브러리 (`stress_test.py`)

**역사 재생** (해당 일자의 실제 종목/팩터 수익률을 현재 포트폴리오에 적용):

| ID | 시나리오 | 일자 |
|---|---|---|
| H1 | 글로벌 금융위기 최악일 | 2008-10-24 |
| H2 | 미국 신용등급 강등 | 2011-08-09 |
| H3 | 코로나 폭락 | 2020-03-19 |
| H4 | 금리 쇼크 | 2022-09-26 |
| H5 | 엔캐리 청산 | 2024-08-05 |

종목 데이터가 해당 일자에 없으면(상장 전) → **팩터 노출 × 당시 팩터 충격**으로 근사한다. 이것이 팩터 분해(3.1)의 실전 용도다.

**가상 시나리오** (역사에 없는 충격 대비):

| ID | 시나리오 |
|---|---|
| S1 | USD/KRW +7% 급등 |
| S2 | 반도체 지수 -15% |
| S3 | KOSPI -8% 시초 갭 (장중 대응 불가 가정) |
| S4 | 보유 최대 비중 종목 하한가 (-30%) |
| S5 | S1 + S2 동시 발생 (복합 충격) |

매일 장전 리포트에 **최악 시나리오 1개와 예상 손실**을 표기한다.

### 4.2 드로다운 디레버리징 사다리 (`drawdown_ladder.py`)

계좌 고점 대비 누적 드로다운(DD)으로 총노출(gross exposure)을 기계적으로 조절:

```
DD  0 ~ -4%   : 노출 100%, 정상 운영
DD -4 ~ -7%   : 노출 70%, 신규 진입 사이즈 50% 축소
DD -7 ~ -10%  : 노출 40%, 신규 진입 금지, 기존 포지션 관리만
DD -10% 초과  : L4 킬스위치 이관
```

**히스테리시스 필수**: 단계 복귀는 한 단계 위 경계보다 1.5%p 회복해야 발동
(예: -7% 단계에서 -4% 단계로 복귀하려면 DD가 -2.5%까지 회복되어야 함).
경계에서 노출이 매일 출렁이는 것을 방지한다.

### 4.3 포트폴리오 변동성 타겟팅 (`vol_targeting.py`)

```
target_vol   = 연 15%
realized_vol = 포트폴리오 일별 손익의 EWMA 변동성 (halflife 20일) × √252
scale        = min(1.0, target_vol / realized_vol)
총노출       = 기본노출 × scale × 사다리 계수(4.2)
```

변동성이 치솟으면 노출이 자동으로 줄어든다 — 알라딘식 "폭풍 전 돛 줄이기"의 가장 단순한 구현.

### 4.4 크라우딩/동질화 모니터 (`crowding.py`)

우리 신호는 수급(외인·기관 흐름) 기반이다. 즉 **같은 신호를 보는 자금과 같이 들어가고 같이 나가게 된다.** 알라딘의 "리스크 동질화" 문제의 우리 버전.

모니터 항목:

```
C1  보유 종목 간 평균 쌍상관 (60일) > 0.70  → "포트폴리오가 사실상 단일 베팅" 경고
C2  VKOSPI 20 초과 + 5일 내 30% 이상 급등   → 레짐 전환 경고
C3  외인 선물 순포지션이 2년 분위수 상/하위 5% → 포지션 쏠림 경고
경고 2개 이상 동시 발생 → 총노출 자동 -20%p (사다리와 별도 적용)
```

### 4.5 VaR 백테스트 — 모델을 의심하는 루프 (`var_backtest.py`)

**알라딘과 우리의 진짜 차이는 모델이 아니라 모델 검증 루프다.**

```
매일: 전일 산출 VaR95 vs 당일 실제 손익 기록 (초과 여부 플래그)
분기: Kupiec POF 검정
      - VaR95 기준 정상 초과율 = 5% (60거래일 중 약 3회)
      - p-value < 0.05로 초과 과다 판정 시 → 모델 보수화:
        ① t-분포 df 6 → 4
        ② EWMA halflife 단축 (변동성 반응 가속)
        ③ 그래도 실패 시 var_limit 자체를 20% 하향
```

## 5. L4 — 킬스위치 (`kill_switch/`) — 협상 불가 영역

**risk/ 엔진과 완전히 분리된 별도 프로세스로 구현한다.** 리스크 엔진에 버그가 있어도, 모델이 전부 틀려도 작동해야 하는 마지막 층이다.

```
K1  일일 실현+평가 손실 ≥ -3.5%   → 전량 청산 + 텔레그램 긴급 호출 + 24시간 거래 정지
K2  누적 DD ≥ -10%               → 전량 청산 + 사람의 명시적 승인까지 무기한 정지
K3  데이터 이상 (가격 NaN, 시세 5분 이상 미수신, API 인증 실패) → 신규 주문 즉시 차단
K4  게이트 로그 없는 체결 감지     → 신규 주문 차단 + Critical 알림
```

구현 규칙:

- 발동 상태는 `state.json`에 영속화 — **서버 리부트로 해제되지 않는다**
- 해제는 수동 명령(텔레그램 확인 코드 입력)으로만 가능. 자동 복구 코드 작성 금지
- 킬스위치 프로세스는 30초 간격 watchdog. 메인 봇 프로세스 생사와 무관하게 동작
- 기존 검수 체크리스트의 Kill Switch 항목과 통합하되, "실제 발동 테스트"를 Phase 1 완료 조건에 포함

## 10. 안티패턴 — 하지 말 것

1. **VaR을 예측으로 착각하지 말 것.** VaR는 "평소 분포에서의 하한 추정"이지 최악이 아니다. 최악은 스트레스 테스트와 킬스위치가 담당한다.
2. **게이트 우회 경로 금지.** "신호가 너무 좋아서"라는 예외 처리를 코드에 넣는 순간 이 엔진은 죽은 것이다.
3. **한도를 백테스트로 최적화하지 말 것.** 한도는 수익 파라미터가 아니라 생존 제약이다. 한도를 풀어서 백테스트 수익이 오르는 것은 당연하며, 그것이 한도를 푸는 근거가 될 수 없다.
4. **킬스위치 자동 복구 금지.** 킬스위치가 발동했다는 것은 시스템이 이해하지 못하는 상황이라는 뜻이다. 이해하지 못하는 상황의 해제를 시스템에게 맡기지 않는다.
5. **저변동 구간에서 한도 완화 금지.** 조용한 시장은 안전한 시장이 아니라 데이터가 위험을 못 보는 시장이다 (알라딘의 한계 그대로).
