## 1. B-1 결론 — 웹 컬럼명 버그 (실측 확정, 봇 데이터 정상)

"성공 301 / 실패 0 / 36% / 평균 +0.0%"는 **봇 데이터 오류가 아닙니다.** signals(2346건·손절 663·평균 7.55%)·scoreboard(오늘 갱신·lose_count 525·avg_return_pct 0.87) 둘 다 값이 살아있습니다.

**진짜 원인 = 웹이 실DB에 없는 키를 읽어 `undefined`→`?? 0`.** 웹과 동일 경로(supabase-py `select('*')`) 실측:
```
board.loss_count     = None   ← 그런 키 없음 → "실패 0"
board.lose_count     = 525    ← 실제 키
board.avg_return     = None   ← 그런 키 없음 → "평균 +0.0%"
board.avg_return_pct = 0.87   ← 실제 키
board.win_rate       = 36.44  ← 키 일치 → 36% 정상
```

### 웹 수정 (정정 — 이거 안 잡으면 엉뚱한 데 고침)
1. `board.loss_count` → **`board.lose_count`**
2. `board.avg_return` → **`board.avg_return_pct`**
3. ★ **`?? 0` 제거만으론 부족**합니다(undefined→null 될 뿐). **컬럼명을 교정해야 525·0.87이 화면에 뜹니다.**
4. migration 파일의 `loss_count`/`avg_return`은 **실DB와 drift**된 것 — `select('*')`가 받는 건 migration이 아니라 실DB라 **실DB가 SSoT**. migration은 별도 정리 권장(인프라 몫).

## 2. B-2 SSoT 매핑 — 확정 (ⓐ)

성공/실패 분류 규칙(close_reason 기준):

| close_reason | 분류 |
|---|---|
| TARGET_HIT, TARGET | ✅ 성공 |
| STOP_HIT | ❌ 실패 |
| TIME_LIMIT, SAR_REVERSAL | **return_pct 부호 판정 (>0 성공 / <0 실패)** ← ⓐ |
| return_pct = 0 (정확히 무승부) | **제외** |
| OPEN (진행중) | 제외 |

- **승률 = 성공 / (성공 + 실패)** (제외분 빼고)
- **평균 = avg(return_pct), 미산출 시 `null`** (0.0 하드코딩 금지)
- **정본 테이블 = `scoreboard` 단일.** `ai_signal_scorecard`는 ML 예측정확도+페이퍼(별도 역할) — 통합하지 말고 **라벨로 구분 표시** 권장.
- 웹의 "성공/실패 기준" 각주 1줄(주린이 신뢰↑) 환영 — 예: *"목표달성=성공, 손절=실패, 시간/추세청산=손익부호로 판정"*.

ⓐ 채택 근거: ⓑ(제외)는 표본의 30%(692건)를 버려 "불리한 건 뺐다" 의혹 소지. 시간청산도 실제 손익이 났으니 승부에 포함하는 게 정직(퀀트봇·웹봇 공동권고).
